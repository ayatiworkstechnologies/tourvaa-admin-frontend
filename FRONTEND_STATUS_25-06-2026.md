## 1. Project Overview

| Item | Details |
| --- | --- |
| Project name | Tourvaa frontend |
| Main purpose | Public website, admin console, and role-based portals for a travel/tour booking platform. |
| Framework | Next.js 16.2.7 App Router |
| UI stack | React 19.2.4, Tailwind CSS 4, lucide-react, Recharts |
| API stack | Axios through `lib/api.ts`, proxied by Next rewrites from `/api/*` to backend `/api/*` |
| Forms | Mostly manual `useState`; `react-hook-form` dependency is installed but not widely used. |
| User roles | Admin/Super Admin, Customer, Supplier, Agent/Reseller, Affiliate |
| Current frontend status | Partially completed. Build passes, many pages exist, but several API integrations and status tests fail. |

Main frontend flow:

1. Public users browse landing pages, tours, blogs, policies, and auth pages.
2. Users log in through `/login`; token is stored in `localStorage`.
3. `AuthProvider` calls `/dashboard/me` and loads role, permissions, menus, and stats.
4. Role layouts redirect users to the correct dashboard.
5. Admin manages users, roles, permissions, tours, bookings, payments, reports, CMS, refunds, notifications, sessions, and partner approvals.
6. Supplier, agent, customer, and affiliate portals provide role-specific dashboards and workflows.

## 8. Authentication Frontend Flow

Login page flow:

1. User submits email and password on `/login`.
2. Page calls `POST /api/auth/login`.
3. Response token is passed to `loginWithToken()`.
4. `loginWithToken()` stores token through `lib/session.ts`.
5. `AuthProvider.refreshSession()` calls `GET /api/dashboard/me`.
6. User is redirected based on `getDashboardPath(roleSlug)`.

Token save method:

| File | Behavior |
| --- | --- |
| `lib/session.ts` | Stores JWT in browser `localStorage` under `tourvaa_token`. |

Logout flow:

| File | Behavior |
| --- | --- |
| `providers/AuthProvider.tsx` | `logout()` clears local storage and routes to `/login` or caller-provided path. |
| Portal layouts | Some call `logout("/")`. |

Protected route handling:

| File | Behavior |
| --- | --- |
| `providers/AuthProvider.tsx` | Global restore and redirect logic. Treats `/customer`, `/agent`, `/supplier`, `/affiliate` as public so their layouts can self-guard. |
| `components/auth/ProtectedRoute.tsx` | Redirects unauthenticated users and shows `AccessDenied` for missing permission. |
| `components/common/ModuleWrapper.tsx` | Page wrapper using `ProtectedRoute`. |

Role-based UI display:

| Role | Behavior |
| --- | --- |
| Admin | Admin pages use backend `dashboard.sidebar_menu` and permissions. |
| Supplier | `app/supplier/layout.tsx` enforces role slug `supplier`. |
| Agent | `app/agent/layout.tsx` enforces role slug `agent-reseller`. |
| Customer | `app/customer/layout.tsx` enforces role slug `customer`. |
| Affiliate | `app/affiliate/layout.tsx` enforces role slug `affiliate`. |

Session expiry handling:

1. Any API call returning 401 triggers the response interceptor in `lib/api.ts`.
2. It calls `POST /auth/refresh-token`.
3. If successful, queued requests are retried.
4. If refresh fails, `hardLogout()` clears token, shows a toast, and redirects to `/login`.

Current issues:

| Issue | Severity | Fix |
| --- | --- | --- |
| JWT stored in `localStorage` | High | Move to secure, httpOnly cookie session design. |
| Portal paths are treated public by global auth | Medium | Keep role layouts, but centralize portal guard behavior for consistency. |
| Page permission map missing newer admin pages | Medium | Add `chatbot`, `cms`, `refunds`, `supplier-payouts`, `tour-approval`, `audit-logs`. |
| Mixed permission slugs | Medium | Replace `update-agents`, `update-suppliers` with canonical dotted permissions. |
